# AoA Automation Opportunity Scan Application Assessment

Date: `2026-04-05`

## Scope

- source bundle: `archive/seed_pack_exports/aoa-automation-opportunity-scan-seed.zip`
- companion ingress summary:
  `archive/seed_pack_exports/aoa-automation-opportunity-scan-seed-summary.md`
- staging note:
  `seed_staging/future/seed_aoa_automation_opportunity_scan_pack.md`
- checked live surfaces in:
  `aoa-skills`, `aoa-playbooks`, `aoa-techniques`, `aoa-agents`,
  `aoa-evals`, and `Dionysus`

Out of scope for this pass:

- a live owner-repo landing in `aoa-skills`
- downstream promotion into `aoa-playbooks`
- publishing new automation-fit techniques in `aoa-techniques`
- federation-wide rollout beyond current repo-local session-harvest installs

## Verdict

- keep the pack in `Dionysus` as staged future prep, not live owner truth
- the seed is materially useful and likely worth applying later
- the first honest owner is `aoa-skills`
- the best first landing is a separate detector skill plus a small
  donor-harvest extract extension, not a broader automation program
- if planted later, the project will gain a much clearer automation-readiness
  intake seam, but it will also take on new technique and checkpoint debt

## Findings

### 1. The missing detector seam is real

The live `aoa-skills` tree now contains the full explicit session-harvest
family:

- `aoa-session-donor-harvest`
- `aoa-session-route-forks`
- `aoa-session-self-diagnose`
- `aoa-session-self-repair`
- `aoa-session-progression-lift`
- `aoa-quest-harvest`

But it does not contain `aoa-automation-opportunity-scan`.

So this pack is not renaming an already-landed owner surface.
It proposes one new detector seam adjacent to a now-live family.

### 2. The seed fits the live session-harvest family honestly

The package does not try to swallow the family.
Its boundary is:

- donor-harvest can notice `automation_candidate`
- automation-opportunity-scan packages readiness into
  `AUTOMATION_OPPORTUNITY_PACKET`
- route-forks can compare automation vs non-automation paths
- self-diagnose can explain blocked automation readiness
- self-repair can emit the smallest prerequisite route
- progression-lift can mention readiness hints without turning them into total
  authority

That is a good fit with the now-live family shape.
It adds one adjacent detector and packet seam rather than trying to replace the
family nucleus or the quest leaf.

### 3. The seed bridges into an owner surface that already exists in aoa-playbooks

`aoa-playbooks` already has explicit automation-seed posture:

- `docs/AUTOMATION_SEEDS.md`
- `config/playbook_composition_overrides.json`
- `generated/playbook_automation_seeds.json`

Those surfaces already say automation seeds are example-only and that schedule
authority stays elsewhere.

That means this pack does not invent a downstream home from scratch.
If planted well, it would become a cleaner upstream feeder into an existing
automation-seed ecosystem rather than a speculative new branch with no owner.

### 4. The minimal path is the honest first move

The package itself recommends a minimal path:

1. add `aoa-automation-opportunity-scan`
2. add `automation_candidate` to donor-harvest outputs
3. add `seed_ready` and `checkpoint_required` fixtures

Compared against live repo state, this is the strongest first move because:

- `aoa-skills` already owns the live family
- the detector is one bounded executable workflow
- the playbook bridge already exists downstream in concept
- the package is careful not to turn detection into scheduling authority
- it avoids landing route-forks or checkpoint changes before a real detector
  exists

So the likely project effect of a good first planting is:

- clearer detection of repeated work that is truly automation-ready
- less vague pressure to "automate more"
- cleaner routing between `skill`, `playbook_seed`, `technique_candidate`,
  `repair_quest`, and `defer`

### 5. Full application would create real technique debt in aoa-techniques

The skill seed names three pending technique seams:

- `AOA-T-PENDING-AUTOMATION-FIT-MATRIX`
- `AOA-T-PENDING-HUMAN-LOOP-TO-SEED-LIFT`
- `AOA-T-PENDING-APPROVAL-SENSITIVITY-CHECK`

That means a clean landing would not end in `aoa-skills`.
It would predictably lead to later extraction pressure in `aoa-techniques`.

The practical consequence is:

- minimal planting adds a useful detector with bounded pending lineage
- later stabilization would demand a second wave of technique publication
- that second wave should follow live use, not precede it

### 6. The main risk is not automation ambition; it is authority drift

The package is careful about several boundaries:

- detection is not schedule authority
- detection is not silent mutation
- recurring scenario candidates belong later in `aoa-playbooks`
- approval-heavy or self-changing routes must cross checkpoint posture
- derived layers stay downstream

So the main danger is not that the seed is boundary-blind.
The main danger is letting the detector be read as permission to automate
directly.

The risks to watch are:

- mistaking one recurring friction loop for real automation readiness
- using `seed_ready` as if it already grants execution authority
- smuggling self-repair or self-upgrade through an automation detector
- letting playbook-seed hints masquerade as live schedules

## What planting would likely lead to

### If planted minimally

The project would likely gain:

- one explicit detector for automation-ready project loops
- a cleaner donor-harvest handoff when automation value is present
- better separation between "there is automation value here" and
  "this should become a skill or playbook seed"
- less temptation to force all recurring work into generic quest residue

This is the best balance between immediate value and bounded complexity.

### If planted more broadly

The project would likely gain:

- a clearer bridge from reviewed-session harvest to playbook automation seeds
- better readiness diagnosis for blocked automations
- later technique extraction around automation fit and approval sensitivity

But it would also likely trigger:

- more `aoa-skills` generated-surface and fixture work
- a new technique backlog in `aoa-techniques`
- checkpoint-law pressure in `aoa-agents`
- proof-hook pressure in `aoa-evals`

So the broader path is plausible, but it should follow stabilization, not lead
it.

## Recommended Application Posture

Keep staged now:

- preserve the pack in `Dionysus`
- treat it as future prep, not owner-repo truth

If later planted:

1. first land `aoa-automation-opportunity-scan` in `aoa-skills` as a separate
   scaffold core skill
2. add `automation_candidate` as a bounded donor-harvest extract kind
3. add fixture coverage for `seed_ready`, `not_now`, and
   `checkpoint_required`
4. only after live use proves the seam, route recurring automation seed
   candidates toward `aoa-playbooks`
5. publish the technique lineage in `aoa-techniques` only after the detector
   stabilizes

Do not do:

- no direct scheduler or background execution authority
- no silent self-change through the detector
- no first-authoring into `aoa-routing` or `aoa-kag`
- no claim that the automation-fit technique lineage already exists
