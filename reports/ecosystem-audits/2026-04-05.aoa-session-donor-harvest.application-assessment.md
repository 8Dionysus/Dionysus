# AoA Session Donor Harvest Application Assessment

Date: `2026-04-05`

## Scope

- source bundle: `archive/seed_pack_exports/aoa-session-donor-harvest.zip`
- staging note: `seed_staging/donor/seed_aoa_session_donor_harvest.md`
- checked live surfaces in: `aoa-skills`, `aoa-techniques`, and `Dionysus`

Out of scope for this pass:

- a full owner-repo landing in `aoa-skills`
- publishing new technique canon in `aoa-techniques`
- cross-repo rollout beyond the proposed first owner

## Verdict

- keep the bundle in `Dionysus` as donor-only staging for now
- the bundle is materially useful and likely worth applying later
- the first honest owner is `aoa-skills`, not a federation-wide rollout
- do not treat it as immediately merge-ready owner truth because its technique
  lineage is still pending

## Findings

### 1. The adjacent skill seam is real

`aoa-session-donor-harvest` does not exist in the live `aoa-skills` tree.
The current skills directory has `aoa-quest-harvest`, but no upstream
`aoa-session-donor-harvest` bundle.

That means the donor bundle is not duplicating an already-landed owner surface.
It preserves a genuinely missing adjacent capability.

Evidence:

- `aoa-skills/skills/`
- bundle source `skills/aoa-session-donor-harvest/SKILL.md`

### 2. The bundle's boundary against `aoa-quest-harvest` is honest

The live `aoa-quest-harvest` skill is intentionally narrow:
it handles repeated reviewed quest evidence and final promotion triage.
The donor bundle proposes a broader upstream step: reviewed session artifact to
donor harvest pack to owner-layer recommendation.

That is a real boundary difference, not a cosmetic rename.
So the recommendation to keep `aoa-quest-harvest` narrow is consistent with the
current owner-repo skill contract.

Evidence:

- `aoa-skills/skills/aoa-quest-harvest/SKILL.md`
- `aoa-session-donor-harvest-recommendation.md`

### 3. The proposed first owner is narrow and plausible

The patch bundle targets real current `aoa-skills` surfaces:
`SKILL_INDEX.md`, skill configs, portable overrides, policy matrix, evaluation
fixtures, snapshots, and a repo-local portable `.agents` export.

The live `aoa-skills` worktree is clean at `1468c00`, and the patch paths match
surfaces that exist now.
So this is a plausible owner-repo-first bundle rather than a vague cross-repo
wish list.

Evidence:

- `aoa-session-donor-harvest.patch`
- `aoa-session-donor-harvest-deliverables.md`
- `aoa-skills` at `1468c00`

### 4. The bundle is not yet ready to count as owner truth

Both technique dependencies remain pending:

- `AOA-T-PENDING-SESSION-DONOR-HARVEST`
- `AOA-T-PENDING-OWNER-LAYER-TRIAGE`

This pass found no published technique surfaces for those IDs in
`aoa-techniques` or `aoa-skills`.
That means the proposed skill can only land honestly as a scaffold with pending
lineage, not as a fully grounded canonical/evaluated skill.

Evidence:

- `skills/aoa-session-donor-harvest/SKILL.md` in the donor bundle
- `skills/aoa-session-donor-harvest/techniques.yaml` in the donor bundle
- checked search across `aoa-techniques` and `aoa-skills`

### 5. The donor bundle respects current AoA boundaries

The proposed skill explicitly keeps `aoa-routing` and `aoa-kag` out of
first-authoring and routes broader donor extraction before handing off to
`aoa-quest-harvest`.
That aligns with the current AoA source-owned vs derivative-layer posture.

So the main risk is not boundary collapse.
The main risk is premature owner-repo landing before lineage and governance are
decided.

Evidence:

- donor bundle `skills/aoa-session-donor-harvest/SKILL.md`
- live `aoa-skills/skills/aoa-quest-harvest/SKILL.md`

## Recommended Application Posture

Keep staged now:

- preserve the bundle in `Dionysus`
- treat it as donor input, not as live `aoa-skills` truth

Apply later, if chosen:

- first land it in `aoa-skills` as an explicit scaffold skill
- keep the `aoa-quest-harvest` narrowing patch in the same bounded change
- only then decide whether the pending technique lineage should be published in
  `aoa-techniques`

Do not do:

- no direct federation-wide rollout
- no direct `aoa-routing` or `aoa-kag` adoption
- no claim that the technique lineage already exists

## Final Read

This bundle is worth keeping and likely worth using.

But the honest current verdict is:
`use later as a staged donor input`, not `apply immediately as owner-approved skill canon`.
