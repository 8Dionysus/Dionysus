## Landed First Slices

### `WS01`

`Agents-of-Abyss` still keeps compact registry v1 bounded, but it no longer
pretends `aoa-sdk` is outside machine-readable federation coverage entirely.
`generated/federation_supporting_inventory.min.json` now carries `aoa-sdk` as a
supporting consumer surface, and center docs route readers to that split
honestly.

### `WS02`

`aoa-sdk` now treats route-directed public surfaces as compatibility-gated only:
`src/aoa_sdk/routing/picker.py` no longer falls back to raw path loads, and the
first latest-sibling canary lane exists in
`.github/workflows/latest-sibling-canary.yml` with an explicit sibling matrix in
`scripts/sibling_canary_matrix.json`.

### `WS10`

`WS10` is no longer only seeded in `aoa-sdk`. `aoa-evals` now carries its own
latest-sibling canary lane in `.github/workflows/latest-sibling-canary.yml`
backed by `scripts/run_sibling_canary.py` and
`scripts/sibling_canary_matrix.json`. That canary proves the current eval proof
surface against latest `aoa-techniques`, `aoa-skills`, `aoa-agents`,
`aoa-playbooks`, `aoa-memo`, `aoa-stats`, and a real `abyss-stack` source
checkout, not a `/srv` runtime mirror.

`Agents-of-Abyss` and `Tree-of-Sophia` now add the next `WS10` layer from the
source side rather than the consumer side. Each repository carries its own
scheduled `source-side-smoke.yml` workflow that runs the current owner-native
validation battery on an unsynchronized schedule. The center and the source
tree now have distinct smoke towers, so mesh expansion no longer depends only
on consumer-facing repos hearing drift first.

`abyss-stack` now adds the next runtime-side `WS10` layer with a dedicated
mirror canary in `.github/workflows/mirror-canary.yml`. That lane materializes
a temporary deployed `Configs` tree from the source checkout, runs
`scripts/aoa-sync-configs --delete`, and then proves
`python scripts/validate_stack.py --parity-check` against the materialized
runtime mirror. This keeps the stack-side source/deployed boundary observable
without treating `/srv/abyss-stack` as an editable source repo.

The remaining `WS10` tail is now closed in `aoa-routing`, `aoa-playbooks`, and
`aoa-kag`. Those repositories already carried compatibility canaries, but they
were synchronized onto the same exact schedule. Their canary lanes are now
desynchronized, and `aoa-routing` was refreshed to current sibling state so the
mesh no longer depends on a known-stale router snapshot. In practice this means
the remaining routing, playbook, and KAG towers can hear drift at different
times instead of failing in one correlated pulse.

### `WS05`

`WS05` no longer stays fully deferred behind the broad old ToS/KAG framing.
The first trimmed follow-through slice is now landed in the two owner repos that
actually own the seam:

- `Tree-of-Sophia` now has a public `python scripts/validate_tiny_entry_route.py`
  surface, and the current docs and workflows now state the route in clearer
  vocabulary: `README.md` stays the public `tos-root`, the worked tiny-entry
  example stays the public compatibility authority surface, and source-first
  re-entry is stated explicitly before any derived `kag_view` or adjunct.
- `aoa-routing` now keeps `tos-root` return navigation bounded in a stronger
  way: the federation root return still restores `Tree-of-Sophia/CHARTER.md` as
  root authority, but it now also publishes one source-owned secondary re-entry
  back to `Tree-of-Sophia/examples/tos_tiny_entry_route.example.json` instead of
  making the ToS return posture end at abstract root authority.

This keeps the seam source-first without replaying the whole older workstream.
What remains open is the second trimmed `WS05` slice in `aoa-kag`, where the
adjunct-budget and subordinate-posture follow-through still belongs.

### `WS06`

`Dionysus` now has a durable remediation ledger, Wave 0 / Wave 1 planning
packets, and a machine-checkable owner-repo reality canary tied to the current
first-slice anchors.

### `WS08` and `WS09`
