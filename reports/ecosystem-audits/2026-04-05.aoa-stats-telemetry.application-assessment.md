# AoA Stats Telemetry Application Assessment

Date: `2026-04-05`

## Scope

- source bundle: `archive/seed_pack_exports/aoa-stats-telemetry-seed.zip`
- companion ingress summary:
  `archive/seed_pack_exports/aoa-stats-telemetry-seed-summary.md`
- staging note:
  `seed_staging/future/seed_aoa_stats_telemetry_pack.md`
- checked live surfaces in:
  `aoa-skills`, `aoa-evals`, `aoa-playbooks`, `aoa-agents`, `aoa-memo`,
  `aoa-techniques`, and `Dionysus`

Out of scope for this pass:

- a live multi-repo telemetry landing
- publishing a new observatory or stats repo
- adding receipt emission to owner repos
- publishing telemetry-specific techniques

## Verdict

- keep the pack in `Dionysus` as staged future prep, not live owner truth
- the seed is materially useful and likely worth applying
- do not plant it as one sovereign cross-repo stats package
- the first honest move is a split owner-local receipt wave led by
  `aoa-skills` and `aoa-evals`
- broader application would likely require a later derived stats or
  observatory-style adjunct rather than stuffing federated truth into current
  owner repos

## Findings

### 1. The first instrumented family already exists

The live `aoa-skills` tree already contains the relevant first-wave family:

- `aoa-session-donor-harvest`
- `aoa-session-route-forks`
- `aoa-session-self-diagnose`
- `aoa-session-self-repair`
- `aoa-session-progression-lift`
- `aoa-automation-opportunity-scan`
- `aoa-quest-harvest`

So the seed is not inventing an abstract target.
It names telemetry around a family that is already real.

### 2. The ecosystem already has evidence-shaped pieces, but not one shared receipt spine

Live surfaces already expose several adjacent ingredients:

- `aoa-skills` donor and automation surfaces already use `session_ref`,
  `evidence_refs`, and bounded packet shapes
- `aoa-playbooks` already carries reviewed-run and automation-seed posture
- `aoa-agents` already has an `agent_progression` schema with evidence refs
- `aoa-memo` already preserves `observed_at` and `supersedes` posture
- `aoa-evals` already carries bounded verdict and evidence-ref structures

But the checked live surfaces do not expose one shared envelope contract for:

- `event_kind`
- `event_id`
- `run_ref`
- `session_ref`
- `actor_ref`
- `object_ref`
- append-only correction through receipt supersession

So the seed fills a real seam.
It does not duplicate an already-landed shared telemetry contract.

### 3. The seed is careful about the right boundary

The package repeatedly keeps one healthy split:

- event facts are not proof
- proof does not collapse into counts
- progression is multi-axis, not one score
- evidence should be linked, not duplicated
- derived summaries should not become owner truth

That is the strongest part of the pack.
It aims at observability without pretending observability owns meaning.

### 4. The main structural gap is owner shape, not value

The pack spans several current owners at once.
`aoa-skills` is the first instrumented workflow family.
`aoa-evals` is the honest home for bounded proof verdicts.
`aoa-playbooks`, `aoa-agents`, and `aoa-memo` each hold later downstream
pieces.

But there is no current dedicated stats or observatory repo in `/srv`.

That means a full planting would force one of two moves:

- awkwardly push derived telemetry authority into existing owner repos
- or create a later dedicated derived layer for cross-repo read models

So the real issue is not whether the pack has value.
It is that the pack has multi-owner shape from day one.

### 5. The first honest landing is phased and split

Compared against live repo state, the best first move is:

1. keep the shared envelope and typed receipt schemas as portable contract
   references
2. instrument the live session-harvest and automation family in `aoa-skills`
   with bounded finish receipts
3. add an `eval_result_receipt` seam in `aoa-evals`
4. delay object summaries, repeated-window summaries, and progression or
   automation rollups until receipt contracts stabilize

This is stronger than trying to land everything at once because:

- the family that most needs receipts is already live
- `aoa-evals` already owns bounded proof meaning
- derived views become much safer once owner-local receipts are real
- it avoids creating a fake sovereign stats empire before any honest
  instrumented sources exist

### 6. Full application would create real cross-owner contract debt

If planted beyond the first receipt wave, the pack would predictably create:

- schema versioning work across multiple owner repos
- append-only correction and supersession policy pressure
- pressure for actor, object, and run reference normalization
- new derived-summary builders and validators
- likely demand for one later stats or observatory-style adjunct

That does not mean the pack is wrong.
It means the pack is infrastructure-shaped, not just skill-shaped.

### 7. The main risk is not counting too much; it is authority drift

The seed is boundary-aware, but the failure mode is obvious:

- counts start behaving like verdicts
- derived summaries start behaving like owner truth
- one aggregate score starts impersonating route or quest progression
- receipts duplicate raw artifacts instead of linking to them
- a central telemetry surface starts making claims that belong in
  `aoa-skills`, `aoa-evals`, or other owners

So the main danger is not telemetry ambition by itself.
It is letting telemetry quietly become governance.

## What planting would likely lead to

### If planted in the first honest wave

The project would likely gain:

- one shared machine-readable receipt posture for the live session-harvest and
  automation family
- clearer evidence-linked counts without pretending counts are proof
- cleaner handoff between owner-local workflow facts and later derived views
- a more stable base for future automation, progression, and fork summaries

This is the best value-to-risk move.

### If planted more broadly

The project would likely gain:

- cross-repo object and window summaries
- route and progression read models
- automation pipeline visibility
- cleaner long-horizon evidence-linked stats

But it would also likely trigger:

- new schema governance pressure across owner repos
- builder and validator work in more than one repository
- pressure for one derived stats or observatory layer
- harder debates about what is fact, what is verdict, and what is only a
  downstream summary

So the broader path is plausible, but it should follow receipt stabilization,
not lead it.

## Recommended Application Posture

Keep staged now:

- preserve the pack in `Dionysus`
- treat it as future prep, not owner-repo truth

If later planted:

1. first instrument `aoa-skills` session-harvest and automation surfaces with
   bounded finish receipts
2. add one bounded `eval_result_receipt` seam in `aoa-evals`
3. keep owner meaning and proof meaning where they already live
4. only after receipt contracts prove stable, add derived read models in a
   downstream stats layer or observatory-style adjunct
5. extract reusable telemetry practice into `aoa-techniques` only after the
   live receipt wave stabilizes

Do not do:

- no sovereign global score
- no replacement of verdicts with counts
- no raw artifact duplication when evidence refs are enough
- no central telemetry surface claiming workflow or proof ownership
- no first-authoring into `aoa-routing` or `aoa-kag`
