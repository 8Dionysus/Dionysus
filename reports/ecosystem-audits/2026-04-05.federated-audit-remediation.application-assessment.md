# Federated Audit Remediation Application Assessment

Date: `2026-04-05`

## Scope

- source pack: `archive/seed_pack_exports/federated_audit_remediation_seed_pack.zip`
- staging note: `seed_staging/audit/seed_federated_audit_remediation_pack.md`
- checked live surfaces in: `8Dionysus`, `Agents-of-Abyss`, `Dionysus`,
  `Tree-of-Sophia`, `aoa-sdk`, `aoa-evals`, `aoa-routing`, and `aoa-skills`

Out of scope for this pass:

- deep repo-local re-audit of every schema seam in `aoa-agents`, `aoa-memo`,
  `aoa-playbooks`, and `aoa-kag`
- dedicated ATM10-Agent product audit
- source-repo-only `abyss-stack` checks outside the `/srv/abyss-stack` runtime
  mirror context

## Verdict

- keep the pack staged in `Dionysus`; do not treat it as a batch-rollout order
  by itself
- the pack is still materially needed, but not uniformly
- highest-need application remains in `WS01`, `WS02`, `WS06`, `WS08`, `WS09`,
  and `WS10`
- `WS05` and `WS11` are only partially needed because live repos already closed
  some of their structural claims
- `WS12` should stay staged until a dedicated ATM10-Agent audit exists

## Findings

### 1. `WS01` remains necessary

`Agents-of-Abyss` still says compact registry v1 intentionally leaves `aoa-sdk`
outside machine-readable inventory scope in `ECOSYSTEM_MAP.md`.
The test suite preserves that exclusion in
`tests/test_validate_ecosystem.py`.
That means the federation still lacks the wider inventory/BOM shape the pack is
asking for, even though the public profile already routes humans toward
`aoa-sdk`.

Evidence:

- `Agents-of-Abyss/ECOSYSTEM_MAP.md`
- `Agents-of-Abyss/tests/test_validate_ecosystem.py`
- `8Dionysus/README.md`

### 2. `WS02` remains necessary

`aoa-sdk` still documents versionless upstream dependency handling in
`docs/versioning.md`, explicitly calling out
`aoa-playbooks/generated/playbook_activation_surfaces.min.json` as an
`unversioned` strict-shape surface.
At the same time, the checked workflows in `aoa-sdk/.github/workflows/` only
run local tests, lint, typing, and package build.
They do not yet publish a sibling matrix or scheduled latest-sibling canary.

This means the control-plane seam is more explicit than before, but the pack's
CI and compatibility hardening asks are still live.

Evidence:

- `aoa-sdk/docs/versioning.md`
- `aoa-sdk/.github/workflows/repo-validation.yml`
- `aoa-sdk/.github/workflows/release-artifacts.yml`

### 3. `WS06` remains necessary

`Dionysus` has the right rule in prose:
owner-repo reality must beat staging truth.
That rule is captured in `docs/codex/owner-repo-reality-check.md`.
But the current `scripts/` directory still has lifecycle validation only; there
is no owner-repo reality canary that checks target surfaces, validator anchors,
or planting state before trusting staged packs.

So the seed garden is better disciplined than before, but it still depends on
manual archaeology for the exact gap this workstream names.

Evidence:

- `Dionysus/docs/codex/owner-repo-reality-check.md`
- `Dionysus/scripts/validate_seed_note_lifecycle.py`
- `Dionysus/scripts/`

### 4. `WS08` and `WS09` remain necessary

`aoa-skills` still shows canonical/default skills whose public surfaces do not
have a matching status-promotion review link.
In both `generated/public_surface.md` and `generated/governance_backlog.md`,
`aoa-change-protocol` and `aoa-tdd-slice` carry `default_reference` posture but
still show `-` for the promotion-review slot.

The checked workflows also do not yet make technique-drift checking or
packaging smoke part of the basic repo-validation lane.
`repo-validation.yml` runs only `python scripts/release_check.py`, while the
portable-export workflow is broader but still keeps `skills-ref` installation
non-blocking via `continue-on-error`.

That is strong evidence that governance truth-sync and mandatory validator-rail
hardening are still worth applying.

Evidence:

- `aoa-skills/generated/public_surface.md`
- `aoa-skills/generated/governance_backlog.md`
- `aoa-skills/.github/workflows/repo-validation.yml`
- `aoa-skills/.github/workflows/codex-portable-export.yml`

### 5. `WS10` remains necessary

The checked federation workflows are still mostly repo-validation towers:

- `Agents-of-Abyss/.github/workflows/repo-validation.yml`
- `Tree-of-Sophia/.github/workflows/repo-validation.yml`
- `aoa-evals/.github/workflows/repo-validation.yml`
- `aoa-sdk/.github/workflows/repo-validation.yml`

`aoa-evals` does pin sibling repos inside repo-validation, which is good, but
the checked workflow set still lacks a separate scheduled canary posture.
`Tree-of-Sophia` and `Agents-of-Abyss` also remain source-local in their
checked workflows.

So the pack's "more even nervous system" claim still matches live repo reality.

Evidence:

- `Agents-of-Abyss/.github/workflows/repo-validation.yml`
- `Tree-of-Sophia/.github/workflows/repo-validation.yml`
- `aoa-evals/.github/workflows/repo-validation.yml`
- `aoa-sdk/.github/workflows/repo-validation.yml`

### 6. `WS05` is only partially needed

The broad structural part of this workstream is no longer fully open.
`aoa-routing` already documents and publishes a source-owned ToS tiny-entry
handoff in `docs/FEDERATION_ENTRY_ABI.md` and
`generated/federation_entrypoints.min.json`.
`Tree-of-Sophia` already owns the current tiny-entry seam and keeps it
source-first.

What still appears open is the lighter follow-through:
source-side smoke/canary coverage and any remaining authority-vocabulary
cleanup.
So `WS05` should not be replayed wholesale; it should be trimmed to the still
unlanded sub-slices.

Evidence:

- `aoa-routing/docs/FEDERATION_ENTRY_ABI.md`
- `aoa-routing/generated/federation_entrypoints.min.json`
- `Tree-of-Sophia/.github/workflows/repo-validation.yml`

### 7. `WS11` is partially needed

The canonical onboarding route is already present in the public profile:
`8Dionysus/README.md` explicitly routes readers to `Agents-of-Abyss`,
`aoa-sdk`, and the other layer homes.
But this pass did not find a distinct public support matrix or clear
federation-wide release-semantics surface in the checked top-level docs.

So the onboarding-path part is already materially landed, while the
support-matrix and release-clarity part still looks open.

Evidence:

- `8Dionysus/README.md`
- `aoa-sdk/README.md`
- checked docs search across `8Dionysus`, `Agents-of-Abyss`, and `aoa-sdk`

### 8. `WS12` should remain staged

The pack itself is honest that ATM10-Agent is a preliminary endcap, not a deep
finished audit.
This pass did not run a dedicated ATM10 reality check, and there is no reason
to let a federation pack silently become the authority for that product-edge
audit.

So `WS12` should remain preserved as a bounded follow-on, not promoted as part
of the first remediation wave.

Evidence:

- `seed_staging/audit/seed_federated_audit_remediation_pack.md`
- `archive/seed_pack_exports/federated_audit_remediation_seed_pack.zip`

## Recommended Application Posture

Apply first, as bounded repo-local slices:

- `WS01`
- `WS02`
- `WS06`
- `WS08`
- `WS09`
- `WS10`

Apply only after a narrower seam re-check:

- `WS03`
- `WS04`
- `WS11`

Treat as partially superseded and trim before use:

- `WS05`

Keep staged until a dedicated audit exists:

- `WS12`

## Final Read

This pack should be kept and used.
It should not be obeyed as a monolithic queue.

The live repos support a selective-application verdict:
the pack still captures real unresolved federation debt, but several later or
broader claims now need trimming against current owner-repo reality before any
landing starts.
