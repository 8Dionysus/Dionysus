# AoA Session Harvest Family Application Assessment

Date: `2026-04-05`

## Scope

- source bundle: `archive/seed_pack_exports/aoa-session-harvest-family-seed.zip`
- companion ingress summary:
  `archive/seed_pack_exports/aoa-session-harvest-family-seed-summary.md`
- staging note:
  `seed_staging/future/seed_aoa_session_harvest_family_pack.md`
- checked live surfaces in:
  `aoa-skills`, `aoa-techniques`, `aoa-playbooks`, `aoa-evals`,
  `aoa-agents`, and `Dionysus`

Out of scope for this pass:

- changing the already-landed owner-repo family
- changing the already-published family techniques in `aoa-techniques`
- widening `aoa-quest-harvest` beyond its current leaf role

## Verdict

- keep the pack in `Dionysus` as landed-upstream lineage, not live owner truth
- the family direction proved materially useful and is already applied upstream
- the first honest owner was `aoa-skills`
- the project now has a richer post-session growth pipeline than the original
  donor-plus-quest contour
- the main continuing debt is now truth-sync, upkeep, and the remaining
  `aoa-quest-harvest` technique backlog

## Findings

### 1. The seed no longer describes a hypothetical family; it now matches a landed owner contour

Live `aoa-skills` now contains:

- `aoa-session-donor-harvest`
- `aoa-session-route-forks`
- `aoa-session-self-diagnose`
- `aoa-session-self-repair`
- `aoa-session-progression-lift`
- `aoa-quest-harvest`

So this pack now functions as lineage for a family that did land, rather than
as an abstract future proposal.

### 2. Planting the family did change post-session work from one harvest step into a small explicit pipeline

The live system now has:

- donor extraction and owner routing through `aoa-session-donor-harvest`
- explicit next-route branching through `aoa-session-route-forks`
- explicit diagnosis and repair surfaces through
  `aoa-session-self-diagnose` and `aoa-session-self-repair`
- explicit progression reflection through `aoa-session-progression-lift`
- final repeated-quest promotion triage through `aoa-quest-harvest`

The landed family now makes named artifacts first-class:

- `HARVEST_PACKET`
- `FORK_CARDS`
- `DIAGNOSIS_PACKET`
- `REPAIR_PACKET`
- `PROGRESSION_DELTA`

The project no longer treats post-session growth as only "extract donor units,
then maybe promote a quest."
It now has explicit surfaces for:

- next-route branching
- failure or drift diagnosis
- bounded repair planning
- evidence-backed progression reflection

This made project growth after reviewed sessions more legible and more
structured, but also more operationally layered.

### 3. The live landing widened from the minimal path into a bounded full-family rollout

The package's minimal path was:

1. expand `aoa-session-donor-harvest`
2. add `aoa-session-route-forks`
3. keep a narrow handoff to `aoa-quest-harvest`

The actual rollout did not stop there.
It expanded into the full bounded family while still preserving two important
constraints:

- `aoa-quest-harvest` stayed narrow
- the new siblings remained explicit, post-session, and review-backed

So the project gained the full family contour without collapsing the quest leaf
into a universal growth skill.

### 4. The growth path did create real technique work in `aoa-techniques`

The seed's predicted seams were in fact extracted and published.
Live family support now includes:

- `AOA-T-0075` `session-donor-harvest`
- `AOA-T-0076` `owner-layer-triage`
- `AOA-T-0077` `harvest-packet-contract`
- `AOA-T-0078` `decision-fork-cards`
- `AOA-T-0079` `risk-passport-lift`
- `AOA-T-0080` `session-drift-taxonomy`
- `AOA-T-0081` `diagnosis-from-reviewed-evidence`
- `AOA-T-0082` `repair-shape-from-diagnosis`
- `AOA-T-0083` `checkpoint-bound-self-repair`
- `AOA-T-0084` `progression-evidence-lift`
- `AOA-T-0085` `multi-axis-quest-overlay`

The only obvious remaining lineage gap is still the quest leaf:

- `AOA-T-PENDING-QUEST-HARVEST`
- `AOA-T-PENDING-PROMOTION-TRIAGE`

### 5. The family did widen the meaning of "project growth" across owner layers

The family manifest routes later candidate outputs toward:

- `aoa-techniques`
- `aoa-skills`
- `aoa-playbooks`
- `aoa-evals`
- `aoa-memo`
- `aoa-agents`

Now that it is planted, the project treats reviewed sessions as a richer source
of:

- reusable workflow candidates
- explicit route decisions
- diagnosis and repair candidates
- progression evidence
- bounded cross-layer owner recommendations

That is useful, and it did change the ecology.
Project growth becomes less "one skill decides a next artifact" and more
"one family emits several review-backed packets that may feed several owner
layers."

### 6. The main risk is no longer premature family inflation; it is upkeep and truth-sync

The landing preserved several important boundaries:

- it keeps everything explicit and post-session
- it keeps `aoa-quest-harvest` narrow
- it keeps `aoa-routing` and `aoa-kag` downstream
- it keeps progression descriptive rather than sovereign
- it keeps self-repair checkpoint-governed rather than silent self-mutation

So the main danger is no longer "too much too early."
The main danger now is:

- stale lineage notes in `Dionysus` diverging from owner truth
- copied install roots drifting from `aoa-skills` exports
- future automation or playbook add-ons over-claiming authority too early
- leaving `aoa-quest-harvest` pending-technique debt unresolved for too long

## What planting led to

The project gained:

- a complete post-session project-growth family in `aoa-skills`
- explicit diagnosis, repair, routing, and progression packets
- published family techniques in `aoa-techniques`
- repo-local rollout copies and a root `/srv/.agents` family install

The project also took on:

- more generated-surface and validation responsibility in `aoa-skills`
- a broader technique canon to keep coherent in `aoa-techniques`
- new downstream follow-on pressure for automation, playbooks, evals, memo,
  and agent-law surfaces

## Recommended Application Posture

Keep as lineage now:

- preserve the family pack in `Dionysus`
- treat it as landed-upstream lineage, not owner-repo truth

Going forward:

1. keep `aoa-skills` and `aoa-techniques` as the source of truth
2. keep `aoa-quest-harvest` narrow
3. return later to publish the quest leaf techniques
4. let downstream repos take only bounded follow-on consequences
5. use `Dionysus` to preserve lineage and assess new seed add-ons against the
   now-live family

Do not do:

- no direct federation-wide rollout of the whole family
- no direct import of `.seed` bundles into live repos unchanged
- no use of progression or diagnosis overlays as hidden routing authority
- no assumption that all four new siblings deserve first-wave landing together

## Final Read

This seed is worth keeping because it now documents how the family became real.

Its effect on the project is already strong:
it turned post-session project growth into a more explicit, multi-packet,
multi-owner workflow family.

The honest current recommendation is:

`keep in Dionysus as lineage, keep owner truth in aoa-skills and aoa-techniques, and return later for aoa-quest-harvest technique closure`.
