## Executive Verdict

- System integrity: `coherent+`, roughly `70%` of the way to dependable
  whole-project integrity.
- Controlled autonomy: `pilot-proven`, roughly `45%` of the way to
  `controlled-autonomy-ready`.
- Current system shape: a strong contract-first federation with one real
  pilot-proven autonomy lane, but not yet a uniformly truthful or uniformly
  deployed autonomy system.
- Main gap: the ecosystem is more mature at the contract and validation layer
  than at the live runtime adoption layer.
- Main risk: `abyss-stack` source, deployed runtime, and trial claims can drift
  far enough apart to overstate what is live.

## 3. Critical Gaps

### Finding 1: `abyss-stack` source/runtime drift is real and already visible

Severity: `P0`

Observed:

- `/home/dionysus/src/abyss-stack/scripts/aoa-llamacpp-pilot` contains a
  `verify` subcommand.
- `/srv/abyss-stack/Configs/scripts/aoa-llamacpp-pilot` does not.
- `/home/dionysus/src/abyss-stack/scripts/aoa-sync-federation-surfaces`
  supports `--check --json`.
- `/srv/abyss-stack/Configs/scripts/aoa-sync-federation-surfaces` does not.
- W6 reports in `Dionysus` state that both landed successfully.

Why this matters for integrity:

- The system currently has more than one "truth" for the same runtime
  capability.
- A reader can mistake a source claim or trial claim for a live deployed fact.

Why this matters for autonomy:

- Controlled autonomy requires trustworthy control surfaces.
- If the operator cannot tell whether `verify` is source-only or live, then
  autonomy claims outrun runtime truth.

### Finding 2: The project lacks an explicit truth model for source, runtime,
trial, and live availability

Severity: `P0`

Observed:

- The ecosystem is strong at source-owned contracts.
- The runtime is strong at evidence capture.
- The missing piece is a first-class contract that says whether a feature is:
  source-authored, deployed, trial-proven, or live-available.

Why this matters for integrity:

- Without this split, the project can look more complete than it is.

Why this matters for autonomy:

- Honest autonomy requires honest availability semantics.
- Trial success is not the same as operator-usable live capability.

### Finding 3: Controlled autonomy is proven only on one narrow promoted lane

Severity: `P1`

Observed:

- The strongest evidence is concentrated around
  `llama.cpp -> langchain-api /run` on `5403` with LangGraph.
- That lane is meaningful, but it is still a bounded pilot lane, not a
  generalized runtime autonomy fabric.

Why this matters for integrity:

- The ecosystem-wide story is broader than the one lane currently proven.

Why this matters for autonomy:

- A narrow pilot is enough for proof-of-direction, not for broad operator trust.

### Finding 4: Several layers are contract-rich but runtime-thin

Severity: `P1`

Observed:

- `aoa-agents`, `aoa-playbooks`, `aoa-memo`, and `aoa-kag` are mature as
  contract surfaces.
- Their live runtime adoption remains selective, advisory, or partial.

Why this matters for integrity:

- The federation is coherent on paper and in validation, but uneven in live use.

Why this matters for autonomy:

- Controlled autonomy depends on these layers not only existing, but actually
  constraining the live route.

### Finding 5: Some authoritative layers are still lighter operationally than
their neighbors

Severity: `P1`

Observed:

- `Tree-of-Sophia` has meaningful tests, but no GitHub workflow.
- `Agents-of-Abyss` is a strong center repo but much lighter in machine-checked
  depth than several lower layers.

Why this matters for integrity:

- Source authorities should not be operationally weaker than the layers derived
  from them.

Why this matters for autonomy:

- Autonomy safety depends on the strongest authorities being stable and easy to
  trust.

### Finding 6: A few repos still require invocation knowledge to validate cleanly

Severity: `P2`

Observed:

- `aoa-techniques` and `Dionysus` pass tests with `PYTHONPATH=. pytest -q`, but
  plain `pytest -q` fails due to import-path assumptions.
- `abyss-stack` source tests are blocked locally by missing runtime deps.

Why this matters for integrity:

- The repos are valid, but not uniformly frictionless to validate.

Why this matters for autonomy:

- This is not a blocker for controlled autonomy, but it does reduce portability
  and contributor trust.

## 4. How Far Away Controlled Autonomy Is

### Current status

The system is not at `controlled-autonomy-ready`.
It is at `pilot-proven bounded autonomy on one promoted lane`.

### What is already proven

- bounded mutation can succeed
- approvals can be reduced without immediate collapse
- pause/resume can be preserved
- post-change validation can gate success
- evidence capture is strong enough to review what happened
- local routing and live health checks are real, not hypothetical

### What is only partially proven

- runtime verify/control posture across source and deployed copies
- cross-layer live use of `aoa-agents`, `aoa-playbooks`, `aoa-memo`, and
  `aoa-kag`
- repair, rollback, and recovery posture as first-class operator interfaces
- deployment truth for newly landed autonomy features

### What is not yet proven

- project-wide controlled autonomy beyond the promoted `llama.cpp + LangGraph`
  lane
- broad autonomy claims that remain truthful across source, runtime, and trial
  layers
- production-grade operator trust where the runtime can be audited without
  checking three different truth surfaces

### Distance estimate

- Distance to dependable system integrity: about one focused hardening wave.
- Distance to controlled-autonomy-ready: about two to three focused hardening
  waves.

Reason:

- the contract scaffolding is already unusually strong
- the pilot lane is already real
- the remaining gap is mostly deployment truth, live adoption depth, and
  operator-control hardening rather than missing conceptual architecture

## Bottom Line

This ecosystem is already stronger than a typical multi-repo AI project at the
boundary, validation, and doctrine layers.
Its main weakness is not lack of architecture.
Its main weakness is that runtime adoption and runtime truth have not fully
caught up with source maturity and trial evidence.

The fastest path forward is not broader expansion.
It is to harden the truth boundary around `abyss-stack`, make the promoted lane
fully operator-trustworthy, and only then widen the autonomy story.

## 2026-04-01 Remediation Addendum

The validator-remediation wave that followed this audit closed the live
questbook and validator defects identified across `aoa-routing`,
`aoa-skills`, `aoa-techniques`, `aoa-evals`, `Agents-of-Abyss`,
`aoa-agents`, and the local-validator dependency surfaces in
`aoa-playbooks`.

The main contract changes are now in place: malformed quest or schema inputs
surface as normal validation failures instead of Python exceptions, additive
quest growth is allowed in `Agents-of-Abyss` and `aoa-agents` while the
foundation quest set remains required, and repositories that rely on `PyYAML`
now declare that prerequisite explicitly through `requirements-dev.txt` and
matching local guidance.

`abyss-stack` remained outside the implementation scope of this remediation
wave. Its source-of-truth checkout remains `~/src/abyss-stack`, while
`/srv/abyss-stack` is treated as a deployed runtime mirror only.
