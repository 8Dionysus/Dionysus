# Seed: Компост контекста

## Суть

ToS не должен быть складом сырого контекста. Он должен быть **пищеварением знания**.

Компост контекста превращает:

* следы действий;
* документы;
* заметки;
* наблюдения;
* фрагменты диалогов;
* ошибки и удачные паттерны;

в слоистую память, из которой можно извлекать не только факты, но и:

* смыслы;
* принципы;
* практики;
* канон под задачу или режим жизни.

Это не архивирование. Это превращение сырья в плодородную почву.

\---

## Человеческое заземление

Для человека, который живёт в режиме **solo+AI**, долго носит тексты в памяти, учится через повторение, телесную практику и возвращение к ядру, память не может быть просто хранилищем. Она должна уметь:

* сжимать без предательства смысла;
* различать временное и долговечное;
* сохранять происхождение;
* давать переносимое ядро для движения, а не только полный архив для кабинета.

ToS должен быть пригоден для кочевого режима, нестабильной среды, фрагментированного внимания и долгих линий становления.

\---

## Проблема

Без компоста контекста память быстро деградирует в одну из двух крайностей:

1. **Свалка**

   * всё сохраняется;
   * ничего не переваривается;
   * retrieval тащит шум;
   * важное тонет в громком.
2. **Музей**

   * остаются только красивые формулировки;
   * исчезают следы происхождения;
   * принципы отрываются от опыта;
   * канон становится декоративным.

В результате система:

* повторно пережёвывает одни и те же инсайты;
* не умеет отличать зрелое знание от свежей гипотезы;
* не держит противоречия;
* не знает, чему пора истечь;
* не превращает опыт AoA в мудрость ToS.

\---

## Предложение

Ввести в ToS **многоуровневый цикл компоста**, в котором каждый фрагмент контекста проходит через слой зрелости.

### Слои

1. **Raw**

   * сырой материал;
   * witness traces, документы, цитаты, заметки, наблюдения, логи, кодовые артефакты.
2. **Note**

   * локальное осмысленное резюме;
   * коротко отвечает: что здесь важно, для чего, с какой уверенностью.
3. **Synthesis**

   * объединяет несколько note-узлов;
   * находит тему, паттерн, напряжение, повтор, различие.
4. **Principle**

   * переносимое правило, ограничение, эвристика или формулировка;
   * обязано знать область применимости и пределы.
5. **Canon**

   * сжатый действующий пакет для конкретного режима;
   * не энциклопедия, а переносимое ядро.

\---

## Компостный цикл

Каждый узел должен поддерживать один или несколько переходов:

```text
Ingest -> Distill -> Cluster -> Extract -> Review -> Canonize -> Revisit / Retire
```

Где:

* **Ingest**: принять сырьё с происхождением;
* **Distill**: сделать локальную заметку;
* **Cluster**: связать с соседними смыслами;
* **Extract**: выделить принцип или практику;
* **Review**: проверить зрелость, противоречия, срок жизни;
* **Canonize**: включить в переносимое ядро;
* **Revisit / Retire**: пересмотреть, понизить, архивировать или удалить из активного канона.

\---

## Инварианты

1. **Ничто не становится каноном напрямую из сырья.**
2. **Каждое сжатие хранит происхождение.**
3. **Уверенность и пределы не теряются при повышении слоя.**
4. **Противоречия не стираются ради красоты.**
5. **Не всё должно жить вечно.** У узлов есть decay и review cadence.
6. **Каждый принцип должен быть проверяемым.** Хотя бы через практику, тест, эксперимент или наблюдение.
7. **Канон должен быть переносимым.** Он имеет ограничение по объёму и назначению.
8. **ToS не зависит от сырого скрытого внутреннего монолога.** Компост работает на наблюдаемых следах, заметках, источниках и резюме.
9. **Человек может остановить, понизить или отменить канонизацию.**
10. **Чувствительные данные редактируются по умолчанию.**

## Правила продвижения между слоями

### Raw -> Note

Продвигать, если:

* материал релевантен текущим темам;
* в нём есть наблюдение, ошибка, решение, вопрос или сигнал;
* есть источник и хотя бы минимальная форма редактирования чувствительных данных.

### Note -> Synthesis

Продвигать, если:

* есть минимум 2 связанных note-узла;
* либо один note повторяется во времени;
* либо обнаружено противоречие, требующее отдельного удержания.

### Synthesis -> Principle

Продвигать, если:

* паттерн повторяется в нескольких контекстах;
* формулировка переносима;
* у неё есть пределы применимости;
* есть хотя бы один способ проверки.

### Principle -> Canon

Продвигать, если:

* принцип нужен сейчас;
* он достаточно ясен для действия;
* он не перегружен контекстом;
* он прошёл review или human approval согласно политике проекта.

### Demotion / Retirement

Понижать или выводить из активного канона, если:

* принцип противоречит новым данным;
* истёк срок релевантности;
* он оказался слишком общим, ложным или непереносимым;
* исчезла практическая ценность.

## Что именно компостировать

В первую очередь принимать в ToS:

* witness traces из AoA;
* заметки по проектам;
* архитектурные решения;
* ошибки, провалы, rollback cases;
* практики, которые реально были исполнены;
* документы, которые формируют язык проекта;
* цитаты и концепты, если они связаны с жизнью системы, а не только с эстетикой.

Не тащить в активный компост без фильтра:

* весь транскрипт подряд;
* весь шум из логов;
* дубли;
* неотрефлексированные эмоциональные всплески как канон;
* случайные формулировки без происхождения.

\---

## Форматы хранения

Минимум поддерживать два слоя представления:

1. **Machine-readable**

   * JSON / JSONL / SQLite / graph adapter;
   * для поиска, кластеризации, promotion rules, evals и сборки канона.
2. **Human-readable**

   * Markdown nodes;
   * для чтения, редактирования, ревью и ручного мышления.

Приоритет: **filesystem-first, graph-ready**.
ToS должен жить и без тяжёлой платформы.

## Где сажать в репозитории

Искать и внедрять в следующих местах:

* memory / knowledge / notes modules;
* ingestion pipelines;
* trace exporters из AoA;
* retrieval / RAG / context builders;
* docs / journals / architecture notebooks;
* review commands, weekly digests, canon exporters;
* eval harness, если есть проверка качества памяти и retrieval.

Если репозиторий не содержит полного ToS, seed сажается частично:

* как schema;
* как promotion rules;
* как exporter;
* как docs contract;
* как bridge между AoA trace и knowledge nodes.

## Срезы внедрения

### Slice 0. Layer schema

Создать базовые типы:

* `CompostNode`
* `SourceRef`
* `DecayPolicy`
* `CanonBundle`

### Slice 1. Ingest adapters

Подключить входы из:

* witness traces;
* markdown notes;
* documents;
* ручных заметок.

### Slice 2. Distillation

Добавить преобразование Raw -> Note с редактированием чувствительных данных и кратким summary.

### Slice 3. Clustering and synthesis

Научить систему связывать note-узлы по теме, паттернам, времени, проекту и противоречиям.

### Slice 4. Principle extraction

Добавить явную форму principle-узла с applicability, limits и practice/test refs.

### Slice 5. Canon builder

Собрать экспорт сжатых canon-bundles под режимы:

* builder
* field
* body
* art
* custom

### Slice 6. Review and decay

Добавить политику review, stale-marking, demotion и retirement.

### Slice 7. AoA bridge

Подготовить двусторонний мост:

* AoA экспортирует traces в Raw;
* ToS возвращает canon bundles, principles и policies как опорный контекст.

## Критерии принятия

Seed считается проросшим, если:

* любой релевантный raw-артефакт может быть принят в компост с source refs;
* существует минимум 3 явных слоя зрелости, а лучше все 5;
* ни один principle не живёт без происхождения;
* канон можно собрать в отдельный bundle с ограниченным объёмом;
* у узлов есть review status и decay metadata;
* contradictions могут быть сохранены как first-class nodes или refs;
* хотя бы один путь `Witness Trace -> Note -> Principle` покрыт тестом;
* есть минимум 1 human-readable markdown export;
* retrieval умеет доставать зрелые узлы, не теряя путь назад к сырью.

\---

## Anti-goals

Не превращать seed в:

* свалку всех когда-либо виденных текстов;
* авто-канонизацию без происхождения;
* красивый knowledge graph без практической работы;
* vendor-locked память на одном сервисе;
* систему, в которой свежесть важнее зрелости;
* систему, в которой канон отрывается от тела, действия и проверки.

\---

## Связь с AoA

AoA производит:

* действия;
* следы;
* ошибки;
* находки;
* решения;
* проверки.

ToS переваривает это в:

* note-узлы;
* синтезы;
* принципы;
* протоколы;
* канон.

Потом ToS возвращает AoA:

* опорные принципы;
* текущий канон;
* активные риски;
* открытые вопросы;
* приоритетные практики.

Этот seed является пищеварительной системой между опытом и мудростью.

\---

## Инструкция для Codex

При посеве в конкретный репозиторий:

1. Найди memory, notes, knowledge, retrieval, docs и trace-related модули.
2. Введи минимальный контракт `CompostNode` без жёсткой привязки к конкретному вендору или БД.
3. Подключи ingest минимум из одного реального источника: желательно из witness traces AoA.
4. Добавь переход Raw -> Note.
5. Добавь хотя бы одну форму synthesis или principle extraction.
6. Добавь metadata: source refs, confidence, applicability, limits, decay, review status.
7. Реализуй экспорт минимум одного `CanonBundle`.
8. Добавь минимум 4 теста:

   * ingest сырья;
   * promotion в note;
   * provenance preservation;
   * demotion или stale review.
9. Обнови docs: что компостируется, как читать узлы, как собирать канон.
10. Если репозиторий маленький, сажай seed в минимальной форме, но не теряй инварианты происхождения, review и проверяемости.
