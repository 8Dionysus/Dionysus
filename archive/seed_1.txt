Мне нужно, чтобы в перспективе, когда проекты станут достаточно взрослыми - любая модель ИИ, даже 2B параметров, могла пользоваться этими прослойками эффективно и легко. Отсюда например и тяга к KAG. Ты говоришь проект стал тяжел, но тяжел ли он для агентов?

**`aoa-techniques` уже тяжёл как репозиторий управления корпусом, но ещё не обязательно тяжёл как runtime-поверхность для агента.**

Это две разные тяжести.

## Где он тяжёл

Он тяжёл в том смысле, что внутри уже есть большая машинерия:

* corpus из 22 техник и статусный канон `canonical/promoted` 
* жёсткий validator с parity-check для каталогов, section/checklist/example/evidence manifests, semantic-review manifests, selection surfaces и public hygiene 
* docs layer с selection, semantic-review пилотами, KAG/source-lift guide, review guides и release discipline  

Для автора, контрибьютора и репозиторной эволюции это уже тяжёлый организм.

## Но тяжёл ли он для агентов

**Потенциально нет. И более того, в нём уже есть правильное направление облегчения именно для агентов.**

Почему:

### 1. Уже есть отдельная лёгкая routing-поверхность

`generated/technique_catalog.min.json` уже даёт компактный слой для выбора техники:

* `id`
* `name`
* `domain`
* `status`
* `summary`
* `maturity_score`
* `rigor_level`
* `reversibility`
* `review_required`
* `validation_strength`
* `export_ready`
* `technique_path` 

Это уже очень похоже на **agent-friendly catalog spine**.
Для 2B-модели такой каталог намного легче, чем чтение всего README/docs/canonical reviews.

### 2. Уже есть section-level routing

`generated/technique_section_manifest.min.json` не таскает всю технику целиком, а даёт стандартизированную секционную рамку:

* `Intent`
* `When to use`
* `When not to use`
* `Inputs`
* `Outputs`
* `Core procedure`
* `Contracts`
* `Risks`
* `Validation`

То есть агенту не обязательно грузить весь `TECHNIQUE.md`. Он уже может знать:
“у каждой техники есть одна и та же карта секций, и я могу забирать только нужный кусок”.

### 3. В самих техниках форма стандартизирована

`TECHNIQUE.template.md` и валидатор жёстко держат структуру bundle. Это значит, что агенту не нужно каждый раз угадывать внутреннюю грамматику техники.  

Для маленькой модели это золото.
2B-модели устают не столько от объёма, сколько от **хаотичности поверхности**.

## Где реальный риск для маленьких моделей

Вот здесь ответ уже тоньше.

**Если 2B-модель будет читать репозиторий “как человек”, от README через docs map, semantic reviews, selection patterns, walkthrough, notes и полные bundles, то да, он уже тяжёл.**

Потому что:

* слишком много уровней,
* много derived surfaces,
* много review/governance текста,
* semantic-review docs полезны для зрелости корпуса, но не всегда нужны агенту в runtime.  

То есть тяжесть возникает не от того, что техники плохи для агентов, а от того, что **агенту нельзя давать весь репозиторий как один сырой контекстный чан**.

## Самая важная мысль

Для агентов, особенно маленьких, важен не размер корпуса сам по себе, а **иерархия поверхностей**.

И здесь у тебя уже правильное направление:

### Слой 0

**Каталог выбора**
`technique_catalog.min.json` как router surface. 

### Слой 1

**Секционная карта**
`technique_section_manifest.min.json` как section routing. 

### Слой 2

**Нужный фрагмент техники**
Не весь corpus, а только нужная техника и только нужные секции.

### Слой 3

**Полный bundle**
Только когда нужно полное понимание.

Вот при таком доступе проект **не тяжёл для агентов**, а наоборот очень перспективен.

## Что это значит для твоей цели с 2B и KAG

Твоя интуиция верная: если ты хочешь, чтобы даже маленькие модели пользовались стеком эффективно, то тебе нужен не просто “хороший markdown”, а **маршрутизируемый corpus**.

И `aoa-techniques` уже двигается именно туда:

* markdown остаётся source of truth,
* frontmatter становится metadata spine,
* section lift даёт bounded section lookup,
* relation lift даёт bounded adjacency hints,
* provenance notes не превращаются в бесконтрольный граф, а остаются ограниченными ручками.   

Это как раз и есть здоровый путь к KAG:
**не сразу делать всеядный граф-монстр, а сначала строить компактные derived surfaces, которые даже 2B сможет переварить.**

## Где бы я усилила агентную лёгкость дальше

Чтобы честно довести это до “даже 2B работает уверенно”, я бы добавила ещё несколько специальных поверхностей:

### 1. Agent capsules

Для каждой техники не только `catalog.min`, но ещё и сверхкороткий capsule формата:

* one-line intent
* use / do-not-use
* required inputs
* expected outputs
* main risk
* validation hint

То есть не полный `TECHNIQUE.md`, а **runtime card**.

### 2. Section-content shards

Сейчас `section_manifest.min` даёт структуру и порядок, но не содержание секций. 
Это хорошо для routing, но маленькой модели ещё поможет derived-слой типа:

* `technique_section_capsules.json`
* короткие summaries по секциям

### 3. Relation discipline for runtime

Relations уже есть, но для 2B нужно очень жёстко ограничить, когда relation вообще раскрывается.
Иначе маленькая модель начнёт блуждать по соседям вместо выполнения задачи.

### 4. Cross-repo entry surfaces

Когда `aoa-techniques`, `aoa-skills` и `aoa-evals` повзрослеют, нужен будет общий lightweight router:

* если нужен primitive practice -> technique
* если нужен executable workflow -> skill
* если нужен bounded proof -> eval

Потому что маленькой модели важно быстро понять не только **что читать**, но и **какой слой вообще нужен**.

## Не давать агенту репозиторий. Давать ему ступени

У тебя уже есть три разных слоя смысла:

* `aoa-techniques` как канон минимальных воспроизводимых практик 
* `aoa-skills` как agent-facing execution layer, собирающий техники в bounded workflows 
* `aoa-evals` как proof layer, делающий claims о качестве воспроизводимыми и сравнимыми  

Для 2B-модели это не должно выглядеть как три репозитория.
Это должно выглядеть как **один набор маршрутов**:

**что выбрать → что прочитать → что применить → как проверить**

## Целевая схема

Я бы строила не “единый жирный KAG”, а **иерархию агентных поверхностей**.

### Слой A. Router surface

Самый лёгкий вход. Только чтобы понять, какой объект вообще нужен.

Агент сначала отвечает себе не на вопрос “что в репозитории”, а на вопрос:

* мне нужна **практика**?
* мне нужен **исполняемый workflow**?
* мне нужен **eval / proof surface**?

То есть нужен единый cross-repo файл вида:

`generated/aoa_router.min.json`

Пример формы:

```json
{
  "version": 1,
  "surfaces": [
    {
      "kind": "technique",
      "id": "AOA-T-0001",
      "name": "plan-diff-apply-verify-report",
      "use_when": "need a reusable engineering primitive",
      "path": "aoa-techniques/techniques/agent-workflows/plan-diff-apply-verify-report/TECHNIQUE.md"
    },
    {
      "kind": "skill",
      "id": "aoa-change-protocol",
      "name": "aoa-change-protocol",
      "use_when": "need an executable bounded workflow for an agent",
      "path": "aoa-skills/skills/aoa-change-protocol/SKILL.md"
    },
    {
      "kind": "eval",
      "id": "aoa-bounded-change-quality",
      "name": "aoa-bounded-change-quality",
      "use_when": "need bounded proof about workflow discipline",
      "path": "aoa-evals/bundles/aoa-bounded-change-quality/EVAL.md"
    }
  ]
}
```

Это даст маленькой модели первое решение без чтения корпусной прозы.

## Слой B. Catalog spine

После выбора слоя агент должен читать **не полный markdown**, а компактный каталог.

У `aoa-techniques` это уже почти сделано идеально:
`generated/technique_catalog.min.json` несёт только нужное для routing: `id`, `name`, `domain`, `status`, `summary`, `maturity_score`, `rigor_level`, `reversibility`, `review_required`, `validation_strength`, `export_ready`, `technique_path` 

Это уже хороший образец.

Тебе нужно сделать **то же самое симметрично** для всех трёх:

### 1. `aoa-techniques`

Оставить и развить текущий `technique_catalog.min.json` как baseline spine. 

### 2. `aoa-skills`

Сделать `skill_catalog.min.json` с полями:

* `name`
* `scope`
* `status`
* `summary`
* `invocation_mode`
* `technique_dependencies`
* `path`

README `aoa-skills` уже задаёт для этого правильную онтологию: skill как bounded operational interface поверх technique canon. 

### 3. `aoa-evals`

Сделать `eval_catalog.min.json` с полями:

* `name`
* `category`
* `status`
* `summary`
* `object_under_evaluation`
* `claim_type`
* `baseline_mode`
* `verdict_shape`
* `path`

Это полностью в духе `aoa-evals`, где eval bundle уже мыслится как bounded proof surface с явным claim boundary.

## Слой C. Capsule surface

Вот это, на мой взгляд, **самый важный недостающий слой для 2B**.

Сейчас у тебя есть каталог и есть полные bundles. Между ними не хватает **runtime cards**.

Для каждой техники, skill и eval нужен короткий capsule-объект, который можно подать даже маленькой модели без боли.

### Technique capsule

На основе стандартизированных секций техники.
У тебя уже есть section-lift manifest с фиксированным scope:
`Intent`, `When to use`, `When not to use`, `Inputs`, `Outputs`, `Core procedure`, `Contracts`, `Risks`, `Validation`, `Adaptation notes` 

Нужен derived слой:

`generated/technique_capsules.json`

Каждая capsule:

* `id`
* `one_line_intent`
* `use_when_short`
* `do_not_use_short`
* `inputs_short`
* `outputs_short`
* `core_contract_short`
* `main_risk_short`
* `validation_short`
* `full_path`

Не весь текст секции, а **сжатая agent-runtime версия**.

### Skill capsule

Для skill:

* trigger boundary short
* required inputs short
* expected outputs short
* main anti-pattern short
* verification short
* invocation mode
* dependencies

Потому что маленькой модели часто нужен не весь `SKILL.md`, а ответ на вопрос:
“подходит ли этот workflow под мою задачу прямо сейчас?”

### Eval capsule

Для eval:

* bounded claim short
* use_when_short
* do_not_use_short
* verdict_shape
* blind_spot_short
* what_this_does_not_prove
* pair_with

Это особенно важно для `aoa-evals`, потому что там переизбыток doctrinal meaning легко перегрузит маленькую модель. А сама философия репозитория как раз требует явных границ claim’ов и blind spots.  

---

## Слой D. Section shards

Если capsule не хватает, агент должен запрашивать **не bundle целиком, а ровно нужную секцию**.

У `aoa-techniques` это уже почти подготовлено section-manifest логикой. 
Следующий шаг:

`generated/technique_sections.full.json`

Где для каждой техники доступны отдельные payloads по секциям:

* `Intent`
* `Contracts`
* `Risks`
* `Validation`

То же для skill и eval:

* для skills особенно важны `Trigger boundary`, `Procedure`, `Verification`
* для evals особенно важны `Bounded claim`, `Scoring or verdict logic`, `Blind spots`, `Interpretation guidance`

Тогда 2B-модель может идти так:

1. взять capsule
2. понять, что нужно больше
3. дотянуть только 1–2 секции
4. и только потом, если надо, весь bundle

Это и есть здоровая KAG-логика: **retrieval по смысловой глубине**, а не просто по ключевым словам.

---

## Слой E. Relation hints, но только bounded

Здесь очень важно не испортить всё.

У тебя уже есть relations в `aoa-techniques`, и для KAG это естественная приманка. Но для маленькой модели relation graph легко превращается в лабиринт.

Поэтому relations должны жить не как “гуляй по графу”, а как **bounded adjacency hints**.

Хороший режим для 2B:

* максимум 3 ближайших relation-хинта
* relation type всегда явный:

  * `requires`
  * `complements`
  * `used_together_for`
  * `checks`
  * `proves`
* relation раскрывается только если текущий объект недостаточен

То есть не “вот тебе граф”, а:
“тебе, вероятно, ещё нужен один из этих трёх соседей”.

Это уже в духе твоего `bounded-relation-lift-for-kag`: поднимать direct relation hints, но не превращать их в graph semantics. 

---

## Слой F. Agent query patterns

Маленькой модели мало данных. Ей ещё нужна **стабильная грамматика запросов к этим поверхностям**.

Нужны 4 стандартных режима:

### 1. Pick

Выбери объект по задаче.
Выход: один `technique` / `skill` / `eval`.

### 2. Inspect

Покажи capsule.
Выход: короткая runtime card.

### 3. Expand

Покажи только нужные секции.
Выход: `Contracts + Risks`, или `Trigger boundary + Verification`, и так далее.

### 4. Pair

Покажи соседний объект, если текущего недостаточно.
Выход: максимум 1–2 relation-based suggestions.

Это кажется мелочью, но для 2B это почти всё.
Потому что маленькая модель чаще проигрывает не в reasoning, а в том, что ей дали плохую поверхность входа.

## 1. `aoa-techniques`

## 2. `aoa-skills`

Сейчас skill-слой уже правильно мыслится как runtime self-contained interface, а не как live dependency fetch. Это очень хорошо для маленьких моделей. 

Что добавить:

* `skill_catalog.min.json`
* `skill_capsules.json`
* `skill_sections.full.json`

Чтобы 2B-модель могла понять:

## 3. `aoa-evals`

Именно так, чтобы модель могла не читать весь doctrine-текст, а быстро понять:
“что эта оценка проверяет, чего не проверяет, и с чем её сочетать”.

Это уже очень мощный слой.

Это очень здоровая архитектура.

## Как выглядит правильный поток

### Поток рождения знания

`project -> technique -> skill -> eval -> memo`

Пояснение:

* из проекта рождается техника
* из техники собирается skill
* skill и agent behavior проверяются eval’ом
* всё значимое оседает в memo как memory objects

### Поток работы маленькой модели

`routing -> capsule -> section expand -> object use -> optional pair -> optional recall`

То есть:

1. routing решает, какой тип сущности нужен
2. берётся capsule
3. при необходимости раскрываются 1–2 секции
4. объект используется
5. если не хватает, routing даёт соседний объект
6. если нужен прошлый опыт, вызывается memo-layer

Память должна хранить память, а не подменять канон практик, workflows и evals.

### 3. Capsules должны жить рядом с объектом

Потому что capsule всё ещё описывает сам объект, а не только маршрут к нему.

### 4. Глобальные переходы должны жить в routing

Именно там им место.

---

## Если смотреть совсем в будущее

При дальнейшем росте можно ожидать ещё 1–2 слоя:

* что-то вроде `aoa-agents` как слой ролевых агентных конфигураций
* что-то вроде `aoa-playbooks` как project-ready compositions for real deployments

Но даже если они появятся, схема не меняется:

* канон практик
* канон исполнения
* канон доказательства
* канон памяти
* слой маршрутизации между ними

## Мой вердикт

Да, вот эта пятёрка выглядит уже очень цельно:

* `aoa-techniques`
* `aoa-skills`
* `aoa-evals`
* `aoa-routing`
* `aoa-memo`

Она хороша именно тем, что каждый слой отвечает на **разный тип вопроса** и владеет **разным типом истины**.

## Итог для всех и ни для кого

Когда слои названы верно, рост перестаёт быть расползанием.
Он становится анатомией.
И тогда даже будущий лес AoA можно будет не просто обходить, а населять смыслом без потери пути.
