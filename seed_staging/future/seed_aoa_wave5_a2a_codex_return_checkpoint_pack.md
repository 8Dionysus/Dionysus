# Seed Note: AoA Wave 5 A2A Codex Return Checkpoint Pack

## Purpose

Prepare one bounded future prep surface for governed A2A child-route work
inside `Dionysus`.

This pack preserves a transplant route for:

- `aoa-summon` as the live bounded summon workflow
- Codex-local child targeting as the default local execution surface
- governed `return` planning for failed or narrowed child runs
- checkpoint-bridge planning that stays subordinate to reviewed closeout
- owner-local publication planning and reviewed memo export planning

It does not create a second role registry, second closeout world, second
workspace install story, or new canonical stats family.

## Why this belongs in Dionysus

`Dionysus` is the seed garden and dispatch layer.
This prep pack belongs here because:

- the source artifact spans many live owner repos rather than one repo-local
  patch
- the live owner homes for recurrence, checkpoint closeout, Codex projection,
  memo writeback, and canonical receipt publication already exist, but the
  governed A2A summon layer has not been landed as one bounded route
- the bundle mixes doctrine notes, helper code, examples, and tests that
  should be adapted into repo-native live surfaces rather than copied blindly
- the route must preserve owner boundaries between summon workflow, helper
  planning, role contracts, routing hints, reviewed closeout, memo writeback,
  stats envelopes, and workspace install doctrine
- the uploaded bundle contains transport noise under `__pycache__/`, `.pyc`,
  and `.pytest_cache/` that should stay visible as non-canonical material
  before planting begins
- in the current workspace, the imported `aoa-sdk` test slice does not yet
  collect because `aoa_sdk.a2a` package scaffolding is incomplete; that gap
  should stay visible before anyone mistakes the bundle for already-landed
  code truth

## Boundary posture to preserve

Read this pack under these route boundaries:

- keep `aoa-skills` as the owner of summon workflow meaning and keep
  checkpoint bridge subordinate to `aoa-checkpoint-closeout-bridge`
- keep `aoa-sdk` as a planning-only helper slice for child targeting, return,
  checkpoint context, reviewed closeout request building, memo export
  planning, and owner publication planning
- keep `aoa-agents` as the owner of role profiles, Codex projection chain,
  quest passports, cohort hints, and self-agent checkpoint contract
- keep `aoa-routing` advisory only and weaker than summon approval
- keep `aoa-playbooks` on scenario-level return posture rather than auto-spawn
  logic
- keep `aoa-evals` on bounded proof wording only
- keep `aoa-memo` reviewed and raw-trace-free
- keep `aoa-stats` canonical-envelope only and downstream from owner receipts
- keep `8Dionysus` on shared-root install doctrine
- keep `Agents-of-Abyss` on short AoA-center doctrine only
- keep `abyss-stack` on runtime publication lanes and runtime-facing helper
  notes
- keep bundled `__pycache__`, `.pyc`, and `.pytest_cache` payloads out of
  canonical planting meaning

## Ownership posture

Read later landings under these owner boundaries:

- `aoa-skills` is the first honest owner for `aoa-summon`, its references, and
  the generated summon snippets
- `aoa-sdk` is the first honest owner for the `a2a/rebase` helper slice,
  examples, and tests
- `aoa-agents` is the first honest owner for any surviving role, projection,
  passport, or self-agent checkpoint fold
- `aoa-routing` is the first honest owner for any surviving return-navigation
  hint fold
- `aoa-playbooks`, `aoa-evals`, `aoa-memo`, and `aoa-stats` are only additive
  owner-local doctrine folds into stronger live notes
- `8Dionysus`, `Agents-of-Abyss`, and `abyss-stack` should receive only narrow
  boundary confirmations where live owner surfaces still leave a real gap
- `Dionysus` preserves the pack and route posture only; it does not move owner
  truth here

## Application posture

Treat this pack as staged future prep work.

The honest landing order, if chosen later, is:

1. land `aoa-skills` `aoa-summon` workflow, references, and generated snippets
2. land `aoa-sdk` `a2a/rebase` helper slice only after repairing the
   `aoa_sdk.a2a` package layer and passing repo-native validation
3. fold only surviving additive boundary notes into `aoa-agents`,
   `aoa-routing`, `aoa-playbooks`, `aoa-evals`, `aoa-memo`, and `aoa-stats`
4. confirm `8Dionysus`, `Agents-of-Abyss`, and `abyss-stack` boundary notes
   only if the owner landings still leave a real gap

When planted later, prefer current live filenames and docs rather than
parallel `wave5_*` namespaces inside owner repos.
