# Seed Note: AoA Wave 5 Portability Regeneration Pack

## Purpose

Prepare the next Codex-plane portability and regeneration wave as one bounded
future prep surface inside `Dionysus`.

This pack preserves a source-owned install hardening route for shared-root
Codex deploy surfaces after the current `/srv` Codex-plane landing is already
real.

It does not invent a new owner for MCP meaning. It hardens regeneration,
rendering, and drift validation for the shared-root `.codex` deploy surfaces
without replacing owner-repo truth.

## Why this belongs in Dionysus

`Dionysus` is the seed garden and dispatch layer.
This prep pack belongs here because:

- the source artifact spans `8Dionysus`, `aoa-sdk`, and the shared-root
  install seam rather than one repo-local patch
- the current public workspace install is real, but the source-owned
  regeneration substrate is not yet landed as one bounded rollout
- the bundle mixes doctrine, manifest inputs, rendered examples, and validator
  ideas that should be adapted into repo-native patterns rather than copied
  blindly
- the route must preserve owner boundaries between shared-root install
  doctrine, workspace discovery, and repo-owned MCP implementations
- the raw bundle contains `__pycache__` and `.pyc` transport noise that should
  stay visible as non-canonical material before planting begins

## Boundary posture to preserve

Read this pack under these boundaries:

- keep `8Dionysus` as the source-owned home for shared-root install doctrine,
  regeneration inputs, renderer, validator, and checked-in generated deploy
  surfaces
- keep `aoa-sdk` on workspace discovery, compatibility, and a narrow
  portability boundary note
- keep MCP implementation ownership in `aoa-sdk`, `aoa-stats`, and `Dionysus`
- keep role meaning in `aoa-agents`
- keep portability as manifest-driven regeneration rather than ad hoc
  hand-editing
- keep `__pycache__` and `.pyc` payloads outside canonical planting meaning

## Ownership posture

Read later landings under these owner boundaries:

- `8Dionysus` is the first honest owner for regeneration doctrine,
  manifest/profile inputs, renderer, validator, and the checked-in generated
  `.codex` deploy surfaces
- `aoa-sdk` is the first honest owner for the portability boundary note about
  discovery versus regeneration
- the sibling workspace root remains the live projection target for the
  generated deploy surfaces
- `Dionysus` preserves the pack and rollout posture only; it does not move
  source-owned install truth here

## Application posture

Treat this pack as staged future prep work.

The honest landing order, if chosen later, is:

1. land `8Dionysus` doctrine plus manifest/profile tree
2. land `8Dionysus` renderer, validator, and regenerated checked-in `.codex/`
3. land the narrow `aoa-sdk` portability note

When planted later, adapt the renderer and validator into the live repo-native
toolchain and ignore raw `__pycache__` and `.pyc` payloads from the uploaded
bundle.

## What this pack is not for

- no MCP server renaming
- no transfer of owner meaning from owner repos into `8Dionysus`
- no replacement of workspace discovery with a second renderer in `aoa-sdk`
- no hand-edited path drift as the primary portability strategy
- no user-global `~/.codex/` authority transfer
- no planting of `__pycache__` or `.pyc` transport noise

## Final rule

Keep this pack because it names the next honest portability route for the
Codex-plane after the first shared-root landing is already real.

But preserve the difference between:

- a staged portability and regeneration pack in `Dionysus`
- source-owned install doctrine and rendering in `8Dionysus`
- workspace discovery and compatibility in `aoa-sdk`
- live workspace-root projection of the generated deploy surfaces
