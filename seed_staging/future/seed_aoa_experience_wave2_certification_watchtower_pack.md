# Seed Note: AoA Experience Wave 2 Certification Watchtower Pack

## Purpose

Preserve the second `aoa-experience-*` planting route inside `Dionysus` while
the mechanics are adapted into owner-native repository surfaces.

This pack covers the second wave only:

- `aoa-experience-certification-forge-seed-v0_4.zip`
- `aoa-experience-deployment-watchtower-seed-v0_5.zip`

The route plants the full v0.4 certification forge and the v0.5 watchtower
contract:

1. experience patch proposal
2. release candidate build
3. regression pack binding
4. certification gate evaluation
5. rollback drill proof
6. operator review
7. versioned assistant release declaration
8. post-release retention watch
9. contract-only watched rollout follow-up

## Source Archives

The source archives are external transport artifacts supplied by the operator:

- `/home/dionysus/Загрузки/aoa-experience-certification-forge-seed-v0_4.zip`
- `/home/dionysus/Загрузки/aoa-experience-deployment-watchtower-seed-v0_5.zip`

manifest was read, and the seed-local validators/tests for v0.4-v0.5 passed in
a temporary extraction. Transport cache noise such as `.pytest_cache/`,
`__pycache__/`, generated seed outputs, and archive-local runtime examples must
stay behind unless an owner repo explicitly accepts an adapted surface.

## Approval and Guard Record

posture. The operator gave explicit approval for the next wave on `2026-04-22`
with:


The approval is treated as permission for bounded code and documentation

## Why This Belongs In Dionysus

`Dionysus` is the seed garden and dispatch layer.
This pack belongs here because:

- the source artifacts span many owner repositories rather than one local patch
- the route must preserve exact external source refs while adapting mechanics
  into repo-native certification, proof, watch, rollback, and review contracts
- the second wave is agent-assisted and needs one durable trace for the owner
  split, guard posture, and validation evidence
- later waves should start from live owner repos and this lineage note, not
  from raw archive copying

## Owner Split

Read the second wave through these owner boundaries:

- `Agents-of-Abyss` owns center law, ordered flow, and authority stop-lines.
- `aoa-agents` owns assistant release and deployment role posture only.
- `aoa-evals` owns bounded verdict/proof bundles, never certification itself.
- `aoa-playbooks` owns certification and watchtower choreography.
- `aoa-memo` owns revision and retention gates only after review.
- `aoa-stats` owns derived release and recurrence summaries only.
- `aoa-routing` owns advisory release, rollback, and incident routes only.
- `aoa-sdk` owns typed helper contracts only.
- `aoa-kag` owns only a small post-release pattern candidate seam.
- `abyss-stack` owns runtime records and dry-run storage posture only after the
  runtime owner accepts that slice.
- `Dionysus` owns source lineage, route discipline, and planting trace only.

activation as a later, separately confirmed risk surface.

## Stop-Lines

Do not use this pack to create:

- a new `aoa-experience` repository
- Codex-issued certification
- Codex-issued release approval
- Codex ring promotion
- assistant self-deployment
- durable rollback execution without operator authority
- stats proof authority
- routing authority over owner review
- SDK-local release verdicts
- memory writeback without a gate
- direct `Tree-of-Sophia` or KAG promotion
- default runtime services, host ports, live scheduling, or hidden deployment
