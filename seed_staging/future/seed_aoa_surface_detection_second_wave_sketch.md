# Seed Note: AoA Surface Detection Second-Wave Sketch

## Purpose

Prepare the `aoa-surface-detection-second-wave-sketch` bundle as one compact
cross-repo follow-on note inside `Dionysus`.

This sketch preserves the next tightening spiral around the first-wave
surface-detection seam without pretending that the second wave is already the
current queue or that it can bypass the missing first-wave owner landing.

## Why this belongs in Dionysus

`Dionysus` is the seed garden and dispatch layer.
This prep note belongs here because:

- the source artifact is a compact design sketch, not a live cross-repo patch
- the sketch depends on a first-wave aoa-sdk seam that is not yet verified
  upstream
- the bundle spans `aoa-sdk`, `aoa-skills`, `aoa-routing`, `aoa-stats`, and
  later family-owner repos at once
- the package is sequencing guidance and boundary discipline, not a mandate for
  immediate downstream mutation
- the owner-repo reality check currently shows no upstream trace of the named
  first-wave or second-wave surfaces

## Second-wave contour to preserve

Read this sketch under these sequencing rules:

- keep `aoa-sdk` as the control-plane seam
- harden `aoa-skills` runtime receipts before inventing wider non-skill
  authority
- keep `aoa-routing` as shortlist assistance rather than semantic ownership
- keep `aoa-stats` descriptive and derived rather than promotive
- add compact family-entry or inspect surfaces in owner repos only where
  repeated use proves the value
- preserve `activated` versus `manual-equivalent` honesty labels across every
  later follow-on

## Ownership posture

Read later landings under these owner boundaries:

- `aoa-sdk` remains the first anchor because second wave assumes first-wave
  detection already exists there
- `aoa-skills` is the first home for richer runtime receipts and any optional
  review helper
- `aoa-routing` may add owner-layer shortlist hints, but not semantic truth
- `aoa-stats` may add cross-window summaries, but not promotion authority
- `aoa-playbooks`, `aoa-evals`, `aoa-memo`, `aoa-agents`, and
  `aoa-techniques` may later add compact inspect surfaces, but only after
  repeated real use proves that need
- `Dionysus` only preserves the sketch and its sequencing posture

## Application posture

Treat this sketch as deferred second-wave guidance.

The honest path, if chosen later, is:

1. land the first-wave `aoa-sdk` seam and observe real reports
2. harden `aoa-skills` receipts and `aoa-routing` shortlist hints
3. strengthen closeout packets in `aoa-sdk`
4. add derived observability in `aoa-stats`
5. only then ask owner repos for compact family-entry or inspect surfaces

When planted later, adapt naming, files, and contract shapes to the live owner
repos rather than copying the sketch verbatim.

## What this sketch is not for

- no immediate cross-repo rollout from `Dionysus`
- no treatment of this sketch as queue truth before first wave lands
- no auto-activation of non-skill surfaces
- no turn of `aoa-routing` into the source of semantic meaning
- no turn of `aoa-stats` into promotion authority
- no requirement that every owner repo add new public objects before second-wave
  value appears

## Final rule

Keep this sketch because it names a useful later tightening pass around surface
detection.

But preserve the difference between:

- a deferred second-wave sketch in `Dionysus`
- a real first-wave landing in `aoa-sdk`
- advisory routing and stats follow-ons
- canonical meaning in the owning repos themselves
